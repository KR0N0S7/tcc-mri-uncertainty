# S4 — Pré-computação das reconstruções E2E-VarNet brain 4×

**Status:** ✅ Concluído em 16/05/2026
**Resultado:** 352/352 volumes processados, 0 erros, ~5.4 GB de outputs
**Dataset gerado:** `kaggle.com/datasets/[usuario-kaggle]/tcc-mri-recons-varnet-brain-4x`

---

## 1. Objetivo

Pré-computar reconstruções da E2E-VarNet pré-treinada (Sriram et al., 2020)
sobre os 352 volumes elegíveis do brain multicoil 4×, gerando os artefatos
`{recon, target, error_map, max_val}` por volume que servem de entrada para o
S5 (quantile network + Conformal Prediction).

A pré-computação é necessária porque o S5 fará dezenas de épocas de treino
sobre os erros de reconstrução — refazer a inferência VarNet a cada época seria
proibitivo. Computar uma vez e armazenar elimina ~95% do custo computacional
total do TCC.

## 2. Decisões metodológicas

### 2.1 Reuso do checkpoint oficial

Optou-se pelo `brain_leaderboard_state_dict.pt` do fastMRI oficial em vez de
treinar do zero ou fine-tunar. Razões:

- O escopo do TCC é **uncertainty quantification**, não reconstrução em si — o modelo de reconstrução é um componente fixo da pipeline.
- O checkpoint oficial ficou em primeiro lugar no fastMRI brain 4× leaderboard (SSIM ~0.966) e é o estado-da-arte amplamente comparável.
- Reprodutibilidade: qualquer leitor pode validar contra o mesmo binário (SHA-256 registrado em `checkpoints/MANIFEST.md`).

| Atributo        | Valor                                                                    |
| --------------- | ------------------------------------------------------------------------ |
| Arquivo         | `brain_leaderboard_state_dict.pt`                                        |
| Tamanho         | 114.2 MB                                                                 |
| SHA-256         | `2cdfdde2fe662f8995316a7ba981d8d3f405d2a5fd82a61bfa1bc4e56062262f`       |
| Parâmetros      | 29,936,966 (~30M)                                                        |
| Hiperparâmetros | `VarNet(num_cascades=12, pools=4, chans=18, sens_pools=4, sens_chans=8)` |

### 2.2 Estratégia de storage

Os 186 GB de k-space multicoil bruto não cabem em um único Kaggle Dataset
(limite 100 GB no free tier). Três opções foram avaliadas:

- **C-1: Hugging Face Hub** — backend único sem cap de tamanho, mas adiciona uma dependência externa e leitura via internet (overhead na primeira passada).
- **C-2: Streaming via URLs autenticadas do fastMRI** — sem hospedagem própria, mas URLs expiram em 24-48h e há rate-limiting do NYU.
- **C-3: Datasets Kaggle fatiados por sequência** — 4 datasets cobrindo AXFLAIR-A, AXFLAIR-B, AXT1, AXT1POST. **Escolhida.**

A C-3 ganhou por três razões:

1. **Latência de IO** — filesystem local da VM Kaggle vs internet via cache.
2. **Menos dependências** — toda infra dentro de um ecossistema só.
3. **Auditabilidade** — banca/orientador podem auditar os datasets diretamente no Kaggle.

### 2.3 Modelo de hardware

GPU local (2-4 GB VRAM) é insuficiente para um forward
da VarNet brain (pico de ~4-6 GB VRAM por slice). Optou-se por **Kaggle T4 ×1**
(16 GB VRAM, gratuito, 30h/semana). Tempo total de pré-computação: **~44 min**
para os 352 volumes.

### 2.4 Schema dos outputs

Decisões granulares aprovadas antes da implementação:

| Decisão     | Escolhido                        | Alternativa rejeitada        | Justificativa                                |
| ----------- | -------------------------------- | ---------------------------- | -------------------------------------------- |
| Organização | `data/recons/{split}/{stem}.npz` | Flat com `split` em metadata | Carregamento por split simples no S5         |
| Error map   | `np.abs(target - recon)`         | `target - recon` (com sinal) | Magnitude é o que o quantile net deve prever |
| Schema      | Tudo num único `.npz`            | Arquivos separados + JSON    | IO atômica, evita inconsistência             |
| Atomicidade | `.tmp.npz` + rename              | Escrita direta               | Crash não corrompe arquivos prontos          |

### 2.5 Reprodutibilidade da máscara

`VarNetDataTransform(mask_func=EquispacedMaskFractionFunc(...), use_seed=True)`
deriva a seed da máscara do `fname.name` (apenas nome do arquivo, sem path).
Isso garante que o **mesmo volume produz a mesma máscara em qualquer máquina**
— a alegação de reprodutibilidade do TCC depende disso. O `varnet_sha256`
salvo em cada `.npz` permite verificar que o checkpoint foi o mesmo.

A classe é `EquispacedMaskFractionFunc` (não `EquiSpacedMaskFunc`, legacy).
A diferença não é cosmética: a versão Fraction usa `floor()` para preservar
a fração exata de linhas centrais; a legacy usa `round()` e introduz 1-2
linhas de diferença. O checkpoint brain 4× foi treinado com Fraction.

## 3. Pipeline (5 passos do S4)

| Passo | Artefato                                | Função                                                                    |
| ----- | --------------------------------------- | ------------------------------------------------------------------------- |
| 1     | `src/models/varnet_loader.py`           | Loader determinístico com SHA-256, eval+no-grad, trata prefixos Lightning |
| 2     | `scripts/download_checkpoint.py`        | Fetch do checkpoint oficial com escrita atômica                           |
| 3     | `src/data/kspace_dataset.py`            | Builder com filtro de splits via `raw_sample_filter`                      |
| 4     | `scripts/sanity_check_varnet.py`        | Gate: SSIM ≥ 0.94 em 5 volumes do val                                     |
| 5     | `scripts/precompute_reconstructions.py` | Pipeline batch resumível, atomicidade por volume                          |

O `notebooks/kaggle_precompute.ipynb` orquestra a execução no Kaggle T4.

## 4. Schema do `.npz` produzido

| Campo             | Tipo    | Shape     | Descrição                                           |
| ----------------- | ------- | --------- | --------------------------------------------------- | -------------- | ----------------------- |
| `recon`           | float32 | (S, H, W) | Reconstrução VarNet, center-cropped                 |
| `target`          | float32 | (S, H, W) | RSS da k-space fully-sampled, mesmo shape           |
| `error_map`       | float32 | (S, H, W) | `                                                   | target - recon | `, magnitude pixel-wise |
| `max_val`         | float32 | ()        | `max(target_vol)`, p/ normalização no S5            |
| `volume_id`       | str     | ()        | Stem do .h5 (ex.: `file_brain_AXFLAIR_200_6002460`) |
| `split`           | str     | ()        | `train` \| `val` \| `cal` \| `test`                 |
| `acceleration`    | int32   | ()        | 4                                                   |
| `center_fraction` | float32 | ()        | 0.08                                                |
| `varnet_sha256`   | str     | ()        | Hash do checkpoint usado                            |

## 6. Achados qualitativos

### 6.1 O outlier do `AXFLAIR_200_6002460`

Este volume apresentou SSIM ~0.89 no sanity, ~7 pontos percentuais abaixo dos
outros 4. Análise slice-a-slice (`scripts/diagnose_sanity_outlier.py`) mostrou
um padrão distintivo:

- **Gradiente monotônico de SSIM** ao longo do eixo Z: slices iniciais (base do crânio) em ~0.87, slices superiores (topo) em ~0.91
- **Std intra-volume** 5-10× maior que dos peers (0.0153 vs 0.0008-0.0030)
- Padrão **não** se repete em outros AXFLAIR (variabilidade volume-específica, não sistemática)

**Interpretação:** variabilidade real do dataset, não bug do pipeline. Causas
prováveis (sem visualização visual, hipóteses):

1. Anatomia complexa da base do crânio — interfaces ar/tecido, sinusites, artefatos de susceptibility.
2. Movimento de paciente nos slices iniciais — AXFLAIR tem aquisição longa, comum a cabeça acomodar nos primeiros segundos.

**Valor para o TCC:** este volume é candidato natural para análise qualitativa
no capítulo de resultados — exemplo concreto de "caso difícil onde a UQ ajuda".

### 6.2 Volumes com `recon_size > kspace_shape`

Aproximadamente 1-2% dos volumes brain têm `attrs["recon_size"]` (do H5)
**maior** que o shape efetivo do k-space — provavelmente por zero-padding
durante a aquisição. A função `T.center_crop(output, crop_size)` falha
nesses casos com `ValueError: Invalid shapes`.

**Solução adotada:** substituir `center_crop(output, crop_size)` por
`center_crop_to_smallest(output, target)`, que crop ambos os tensores para
o menor dos shapes por dimensão. Resultado: recon e target sempre saem
com shape idêntico para o cálculo de error_map.

Este detalhe não é documentado no `run_pretrained_varnet_inference.py`
oficial do fastMRI — o script oficial tem o mesmo bug latente, que só não
aparece se o usuário processar apenas os splits canônicos val/test, sem
incluir volumes do train que tem essa propriedade.

## 7. Como reproduzir

### 7.1 Pré-requisitos

- Conta Kaggle Free com GPU T4 disponível
- Repositório clonado em ambiente local com `.env` configurado
- 352 volumes `.h5` em `\data\mri\anotados\` (ou equivalente)
- 4 Kaggle Datasets brain privados (criados a partir das pastas locais)

### 7.2 Sequência local (smoke + validação)

```bash
# 1. Baixar checkpoint e validar SHA-256
python scripts/download_checkpoint.py
# Esperado: SHA-256 = 2cdfdde2fe662f8995316a7ba981d8d3f405d2a5fd82a61bfa1bc4e56062262f

# 2. Rodar testes (devem ser 34+ verdes)
python -m pytest tests/ -v

# 3. Smoke test do pipeline (1 volume, CPU ou GPU local)
python scripts/precompute_reconstructions.py --split val --max-volumes 1 --device cuda

# 4. Sanity check formal (5 volumes do val)
python scripts/sanity_check_varnet.py --use-cache

# 5. (Opcional) Diagnóstico slice-a-slice
python scripts/diagnose_sanity_outlier.py
```

### 7.3 Pré-computação em escala (Kaggle T4)

1. No Kaggle, **Create → New Notebook → Import** → `notebooks/kaggle_precompute.ipynb`
2. **Settings:** Accelerator = GPU T4 ×1, Internet = On
3. **Add Input → Datasets:** os 4 datasets brain (`fastmri-brain-*`)
4. **Run All** — esperado ~44 min total
5. Quando completar:
   - **Output → Create Dataset** → nome `tcc-mri-recons-varnet-brain-4x`, Privado
   - Esse dataset (~5.4 GB) é o input do S5

### 7.4 Verificação pós-execução

```python
# No Kaggle, após Run All:
from pathlib import Path
RECONS = Path('/kaggle/working/recons')
for split in ['train', 'val', 'cal', 'test']:
    n = len(list((RECONS / split).glob('*.npz')))
    print(f'{split}: {n} volumes')
# Esperado: train: 213, val: 46, cal: 46, test: 47
```

## 8. Próximos passos (S5)

Com `data/recons/` pronto, o S5 pode trabalhar exclusivamente sobre os 5.4 GB
de outputs, sem precisar tocar em k-space ou na VarNet:

- Treino de um quantile network (CNN, U-Net, ou similar) para prever os quantis
  superiores do `error_map` a partir do `recon`
- Implementação do CQR (Romano et al., 2019) usando o split `cal` para calibração
- Avaliação de cobertura nominal no `test` (target: 90% de cobertura)
- Análise por modalidade (AXFLAIR, AXT1, AXT1POST) e por região anatômica (slices da base vs topo do crânio)
