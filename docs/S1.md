## 3. Decisões e o raciocínio por trás de cada uma

### 3.1 Plataforma de compute: Kaggle T4 como principal

**Problema.** O projeto envolve treinos de regressão quantílica conformizada sobre saídas de uma VarNet pré-treinada. Mesmo congelando a VarNet, o módulo de incerteza precisa ser treinado, validado e avaliado em GPU. O laptop local não tem GPU dedicada.

**Opções consideradas:**

- **Kaggle T4** (~30 h/semana, free tier): sessões de até 12 h sem interrupção, GPU T4 com 15 GB VRAM, integração nativa com datasets hospedados na plataforma.
- **Google Colab** (free e Pro): T4 disponível no free com timeouts agressivos e desconexões; Pro custa ~USD 10/mês.
- **AWS/Azure cloud**: capacidade superior mas custo proibitivo para um TCC sem financiamento.
- **GPU pessoal**: não disponível; comprar uma RTX 4060/4070 levaria ~R$ 3-5k.

**Decisão.** Kaggle T4 como plataforma principal de compute, Colab como contingência em caso de mudança de política.

**Justificativa.**

- 30 h/semana é compatível com o orçamento de GPU estimado para o projeto inteiro (cerca de 600 h totais distribuídas em ~5 meses).
- Sessões longas são essenciais para treinos de 100k+ iterações.
- Free tier evita variabilidade orçamentária ao longo da fase.

**Risco aceito.** Mudança de política do Kaggle pode reduzir cota. Plano B: migrar para Colab Pro.

### 3.2 Workflow Kaggle (compute) + GitHub (source of truth) + VS Code local (edição)

**Problema.** Iteração rápida sobre código exige edição confortável (autocomplete, refactoring, debug local). Execução exige GPU. Versionamento exige hospedagem central.

**Opções consideradas:**

- **Tudo no Kaggle (notebooks)**: edição precária, versionamento difícil, código fragmentado em células.
- **Tudo local com sync manual**: lento, propenso a erro humano.
- **Cloud IDE (GitHub Codespaces, Gitpod)**: limitações severas no free tier; sem GPU.
- **Notebooks Kaggle puxando de Git**: ideal — código limpo no repo, notebooks são entrypoints finos que apenas importam e executam.

**Decisão.** Tripé:

- **GitHub**: single source of truth para todo código `.py`. Estado canônico do projeto.
- **VS Code local**: edição diária, refactoring, testes unitários sobre CPU, commits.
- **Kaggle**: execução de notebooks que clonam o repo no início e fazem `pip install -r requirements.txt`.

**Justificativa.** Separa as preocupações: o que muda mais (código) fica onde edita melhor (VS Code); o que demanda mais recurso (treino) fica onde tem o recurso (Kaggle). GitHub é o ponto de encontro.

**Operacionalização.** Notebooks no Kaggle começam com `!git clone https://github.com/KR0N0S7/tcc-mri-uncertainty.git`, depois `%cd tcc-mri-uncertainty` e `!pip install -r requirements.txt`. Todo código é importado de `src/`.

### 3.3 Versões pinadas em `requirements.txt`

**Problema.** Pip por padrão instala a versão mais recente compatível. Em meses, isso quebra reprodutibilidade — uma versão patch pode introduzir comportamento diferente em funções numéricas (sklearn, scipy).

**Decisão.** Pin estrito (`==`) em todas as dependências:

```
torch==2.10.0
torchvision==0.18.0
numpy==1.26.4
h5py==3.11.0
pandas==2.2.2
scikit-learn==1.5.0
matplotlib==3.9.0
tqdm==4.66.4
pytest==9.0.3
fastmri @ git+https://github.com/facebookresearch/fastMRI.git
```

**Justificativa.** Conforme Wilson et al. (2017), pinning é uma das _good enough practices_ mais baratas. Custo: zero. Benefício: reprodutibilidade no longo prazo.

**Como auditar.** `pip freeze` em um ambiente recém-criado a partir do `requirements.txt` deve retornar a mesma lista.

**Caveat futuro.** Versões pinadas podem ter security advisories ao longo do tempo. Antes de cada entrega formal (4ª e 5ª), revisitar o `requirements.txt` e atualizar onde aplicável, documentando a mudança.

### 3.4 OneDrive evitado para o repositório local

**Problema.** OneDrive sincroniza arquivos em segundo plano. Para repositórios Git, isso pode corromper o diretório `.git/` quando o OneDrive tenta resolver conflitos em arquivos binários intermediários (objetos do Git, índice, packfiles). A corrupção é tipicamente silenciosa — o repo fica num estado inconsistente que só aparece em operações específicas.

**Decisão.** Manter o repositório local fora da pasta sincronizada pelo OneDrive. Caminho adotado: `C:\...\tcc-mri-uncertainty\`.

**Justificativa.** Corrupção do `.git/` é catastrófica e silenciosa. O backup remoto é o `git push` para o GitHub, feito ao final de cada sessão de trabalho — isso já cobre o risco de perda local.

**Custo.** Sem backup automático contínuo. Mitigado por push diário para o remote.

## 4. Validação executada

Os três sanity checks abaixo foram executados ao término da fase para confirmar que o ambiente está apto a hospedar o trabalho dos próximos meses.

### 4.1 Sanity check da VarNet pré-treinada

Procedimento: carregar o checkpoint `varnet_brain_4x.pt` distribuído pelo fastMRI oficial e executar forward pass sobre entrada sintética.

**Resultado.** Modelo carregou sem erro; output produzido com shape compatível com o esperado para reconstrução acelerada 4× no fastMRI brain.

**Limitação.** Validação semântica (SSIM ≈ 0.966 contra ground truth) não foi feita nesta fase. Está prevista para o início da S4, quando o pipeline de avaliação contra dados reais estiver montado.

**Detalhes operacionais não registrados em tempo real.** O notebook exato e a saída numérica não foram salvos. Para reproduzir, executar `notebooks/01_setup_sanity_check.ipynb` (se versionado) ou recriar o teste manualmente conforme descrito em Sriram et al. (2020).

### 4.2 Determinismo multi-run

Procedimento: rodar a mesma entrada sintética pela VarNet múltiplas vezes consecutivas com seeds fixadas (`torch.manual_seed(42)`, `torch.cuda.manual_seed_all(42)`, `torch.backends.cudnn.deterministic = True`), comparando os outputs.

**Resultado.** Outputs idênticos a precisão de ponto flutuante em execuções consecutivas, confirmando configuração determinística adequada.

**Importância.** Determinismo é pré-requisito para auditoria — diferentes execuções com a mesma seed devem produzir os mesmos resultados.

**Caveat.** Determinismo entre máquinas diferentes (CPU vs GPU, ou GPUs de modelos distintos) não é garantido mesmo com `cudnn.deterministic`. As discussões de reprodutibilidade no TCC devem mencionar essa limitação.

### 4.3 Ausência de memory leak

Procedimento: forward pass em loop sobre a VarNet com entradas sintéticas, monitorando `torch.cuda.memory_allocated()` ao longo de várias iterações.

**Resultado.** Após estabilização inicial nas primeiras iterações, o uso de memória manteve-se constante — sem crescimento monotônico que indicaria leak.

**Importância.** Treinos de 100k+ iterações em GPU com 15 GB de VRAM não toleram leaks. Se a memória crescesse linearmente, o treino seria abortado por OOM antes de convergir.

## 5. Limitações e riscos de auditoria

**(a) Documentação retrospectiva.** Esta fase não foi documentada em tempo real. Outputs específicos dos sanity checks, mensagens de erro encontradas, e o caminho exato até a configuração final não estão registrados. Mitigação: o estado final (repositório atual) é o que importa para o trabalho prosseguir; o histórico operacional não é necessário para auditoria do método.

**(b) Sanity check da VarNet foi sintético.** Forward pass em entrada zerada confirma que o modelo carrega e produz output, mas não que o output é semanticamente correto. A validação semântica (SSIM ≈ 0.966) está deferida para S4.

**(c) Determinismo entre máquinas não verificado.** Configurações `cudnn.deterministic = True` garantem determinismo dentro da mesma máquina, mas resultados podem diferir entre GPUs diferentes (T4 vs A100, por exemplo). Mitigação: documentar no TCC que os experimentos foram feitos sempre no Kaggle T4 para consistência.

**(d) Versões pinadas em `requirements.txt`.** Podem se tornar obsoletas ou ter vulnerabilidades reportadas ao longo dos meses. Mitigação: revisar antes de cada entrega formal.

**(e) Dependência do Kaggle.** Mudança de política da plataforma afetaria o cronograma. Mitigação: plano B em Colab Pro documentado.

## 6. Como reproduzir

**Pré-requisitos:**

- Windows 10/11, Linux ou macOS
- Anaconda ou Miniconda instalado
- Git instalado
- Conta no Kaggle (para acesso à GPU; opcional se reproduzindo só local)

**Setup local:**

```bash
# 1. Clonar repositório
git clone https://github.com/KR0N0S7/tcc-mri-uncertainty.git
cd tcc-mri-uncertainty

# 2. Criar ambiente Conda dedicado
conda create -n tcc python=3.12 -y
conda activate tcc

# 3. Instalar dependências pinadas
pip install -r requirements.txt

# 4. Verificar Python e PyTorch
python -c "import torch; print(torch.__version__, torch.cuda.is_available())"
# Esperado: 2.10.0 True (se GPU local) ou 2.10.0 False (CPU only)
```

**Configuração do Kaggle (para uso de GPU):**

1. Criar conta gratuita em https://www.kaggle.com/
2. Settings > Account > Create New API Token → faz download de `kaggle.json`
3. Mover `kaggle.json` para:
   - Linux/Mac: `~/.kaggle/kaggle.json` + `chmod 600 ~/.kaggle/kaggle.json`
   - Windows: `C:\Users\<usuario>\.kaggle\kaggle.json`
4. Verificar: `kaggle datasets list` (deve listar datasets disponíveis)

**Sanity check da VarNet (reproduzir esta fase):**

Não há um script reservado para isso no repositório atual; a validação foi feita interativamente em notebook não versionado. Para reproduzir o equivalente:

```python
import torch
from fastmri.models import VarNet

# 1. Carregar arquitetura
model = VarNet()

# 2. Tentar carregar checkpoint (precisa ser baixado primeiro - ver S4)
# state = torch.load('checkpoints/varnet_brain_4x.pt')
# model.load_state_dict(state)

# 3. Forward pass sintético
torch.manual_seed(42)
x = torch.randn(1, 1, 4, 320, 320, 2)  # batch, coils, slices, H, W, complex
mask = torch.zeros(1, 1, 4, 320, 320, 1)  # mask completo
mask[..., ::4, :, :] = 1  # subamostragem 4×
y = model(x, mask)

# 4. Determinismo: rodar 3 vezes, comparar
y_runs = [model(x, mask) for _ in range(3)]
assert all(torch.allclose(y_runs[0], y_i) for y_i in y_runs[1:])

# 5. Memory leak: 50 iterações
import gc
mems = []
for _ in range(50):
    _ = model(x, mask)
    gc.collect()
    torch.cuda.empty_cache()
    mems.append(torch.cuda.memory_allocated())
# Memória deve estabilizar nas primeiras iterações
```
