## 3. Decisões e o raciocínio por trás de cada uma

### 3.1 Quais splits do fastMRI baixar

**Problema.** O fastMRI brain multicoil disponibiliza três splits: `train` (~135 GB compressed em 9 tarballs), `val` (~30 GB compressed) e `test` (sem ground truth, apenas para o desafio público). O fastMRI+ anota tanto `train` quanto `val` do fastMRI base.

**Opções consideradas:**

- **Apenas `multi_train`** (135 GB): cobre a maior parte das anotações.
- **`multi_train + multi_val`** (~165 GB): conjunto completo de anotações.
- **Tudo + `test`** (~200 GB): inviável porque `test` não tem ground truth.

**Decisão.** Apenas `multi_train`. O `multi_val` fica como possível extensão futura.

**Justificativa.**

- 135 GB já desafia a infraestrutura local (~200 GB livres no `D:\`).
- O `multi_train` contém a maioria das anotações fastMRI+ (~80%, estimativa baseada na contagem de linhas do `brain.csv`).
- Os 352 volumes eventualmente elegíveis (após filtro em S3) são suficientes para o desenho experimental planejado, com poder estatístico adequado em todas as faixas de tamanho de lesão (Seção 3.3 do S3.md).
- Acrescentar `multi_val` seria uma análise de robustez razoável mas não muda o desenho experimental principal.

**Como auditar.** O `brain.csv` lista todos os volumes anotados pelo fastMRI+. Comparar a lista de `Filename` únicos com o conteúdo de `\data\mri\anotados\` revela quais estão presentes (350 com bbox + 398 study-level = 748, mais 2 que aparecem múltiplas vezes; total 750 arquivos físicos).

**Caveat.** Esta decisão exclui do estudo qualquer paciente do split de validação do fastMRI. Não é um problema metodológico (val/cal/test internos são feitos sobre os 352 do train) mas é uma limitação de cobertura populacional do estudo.

### 3.2 Estratégia de download sequencial vs paralelo

**Problema.** Os 9 tarballs do `multi_train` somam ~135 GB compressed. Após extração, cada `.h5` ocupa entre 200 MB e 1.5 GB. O dataset descompactado completo ocuparia ~280 GB. O `D:\` tem ~200 GB livres. Baixar tudo de uma vez é inviável.

**Opções consideradas:**

- **Paralelo**: baixar todos os 9 tarballs simultaneamente, extrair todos, depois filtrar. Requer ~270 GB de pico (135 GB tarballs + 135 GB extraídos brutos antes da deleção).
- **Sequencial com filtro on-the-fly**: para cada tarball: baixar → extrair → filtrar para `\data\mri\anotados\` → deletar tarball e arquivos não selecionados. Pico de uso ≤ 30 GB.

**Decisão.** Estratégia sequencial.

**Justificativa.**

- Pico de uso fica abaixo do limite do disco.
- Permite que erros em um tarball não comprometam o trabalho já feito.
- Inconveniência aceitável: o tempo total não é significativamente maior, dado que o gargalo é largura de banda da internet, não disco.

**Operacionalização.** Loop manual sobre os 9 tarballs (sem automação completa em um script único; cada tarball foi feito como uma operação separada). Etapas por tarball:

```bash
# Para cada tarball N de 1 a 9:
wget URL_TARBALL_N   # URL assinada, válida por 24h
tar -xf brain_multicoil_train_part_N.tar
# Mover/copiar para \data\mri\anotados\ os .h5 que aparecem no brain.csv
# Deletar o tarball + arquivos não selecionados
```

**Detalhes operacionais não registrados em tempo real.** A automação exata do filtro durante a extração não está em um script único versionado. O `scripts/filtra_anotados.py` versionado hoje é a versão refinada que opera sobre uma pasta de extração já existente, não a versão usada durante o download original.

### 3.3 Filtro pelo `brain.csv` durante o pipeline

**Problema.** Dos volumes extraídos de cada tarball, apenas uma fração tem anotação no fastMRI+. Manter os não-anotados desperdiça disco sem contribuir para o estudo.

**Opções consideradas:**

- **Extrair tudo, depois filtrar**: simples mas usa mais disco intermediário.
- **Filtrar durante a extração**: usa menos disco intermediário, mas exige scripting mais cuidadoso.

**Decisão.** Filtrar logo após a extração de cada tarball, antes de passar para o próximo.

**Justificativa.** Combinada com a estratégia sequencial da Seção 3.2, manter apenas os anotados reduz o pico de uso e o footprint final.

**Critério de filtro.** Volumes cujo `stem` (nome do arquivo sem `.h5`) aparece em pelo menos uma linha do `brain.csv` foram retidos. Resultado: **750 volumes** copiados para `\data\mri\anotados\`.

**Verificação de integridade tripla.** Para cada `.h5` copiado, três checks foram aplicados:

1. Tamanho > 100 KB (descartar arquivos truncados).
2. `h5py.is_hdf5(file)` retorna `True` (formato HDF5 válido).
3. Key `reconstruction_rss` presente no arquivo (estrutura fastMRI esperada).

**Resultado.** Todos os 750 passaram nas três verificações. Nenhum descarte por integridade.

### 3.4 Bug do `brain_df['file']` vs `brain_df['Filename']`

**Problema encontrado.** O guia inicial do projeto (`guia_tcc_mri_incerteza_v2_10de10.docx`) referia-se à coluna do `brain.csv` como `brain_df['file']`. Tentativas de leitura com esse nome falhavam com `KeyError`.

**Causa.** A coluna real no `brain.csv` distribuído pelo fastMRI+ é `Filename` (com `F` capitalizado). O guia tinha um erro de redação.

**Decisão.** Corrigir o nome da coluna em todos os scripts e seguir adiante.

**Justificativa.** Convenção do dado distribuído oficialmente prevalece sobre erro de transcrição no guia.

**Implicação para auditoria.** Quem seguir o guia literalmente vai falhar no primeiro acesso ao `brain.csv`. Todos os scripts hoje versionados (`scripts/filtra_anotados.py`, `scripts/audit_dataset.py`, `scripts/make_splits.py`, etc.) usam `Filename` corretamente.

**Como auditar.** Inspecionar o header do `brain.csv` com `pd.read_csv(brain_csv).columns.tolist()` deve retornar `['Filename', 'Slice', 'Study Level', 'Label', 'X', 'Y', 'Width', 'Height']` (ou similar — verificar contra a versão atual do fastMRI+).

### 3.5 Localização dos artefatos em disco

**Decisão.** Adotar uma estrutura padrão em `\data\mri\`:

```
\data\mri\
├── anotados\           # 750 .h5 filtrados (saída desta fase, entrada para S3)
├── brain.csv           # Anotações fastMRI+
└── multi_train_old\    # Pasta temporária com extração bruta (deletada após filtragem)
```

**Justificativa.**

- Separar dados dos repositório de código.
- `D:\` tem mais espaço e não é sincronizado por nenhum serviço de nuvem.
- Estrutura plana facilita pathing nos scripts (paths absolutos curtos).

**Como auditar.** `dir \data\mri\` deve listar `anotados`, `brain.csv` (e o que mais o usuário tenha decidido manter localmente).

## 5. Limitações e riscos de auditoria

**(a) Documentação retrospectiva.** Como a S1, esta fase não foi documentada em tempo real. Sequência exata de comandos, mensagens de erro encontradas e tempo gasto por tarball não estão registrados.

**(b) Filtro permissivo.** O critério "volume aparece no `brain.csv`" deixou passar 398 volumes apenas com anotação study-level (sem coordenadas). Esses são úteis apenas como controles ou para classificação volume-wise, não para o desenho de incerteza pixel-a-pixel deste TCC. Refilter feito em S3.

**(c) Sem verificação de checksum vs versão oficial.** Os tarballs do fastMRI são distribuídos com URLs assinadas, sem checksums anexados explicitamente nesta fase. Mitigação parcial: a integridade local foi verificada com `h5py.is_hdf5` e presença de keys esperadas (Seção 4.2).

**(d) URLs assinadas expiram em 24h.** Reproduzir a fase requer novo cadastro no fastMRI e obtenção de URLs frescas. Não é possível distribuir as URLs.

**(e) Apenas split de treino do fastMRI.** Limitação de cobertura populacional documentada na Seção 3.1.

**(f) Sem versionamento dos artefatos de dataset.** Os 750 volumes em `\data\mri\anotados\` não estão versionados (impraticável por tamanho). Mitigação: o critério de filtro está documentado neste relatório e o `brain.csv` está disponível publicamente; qualquer pessoa pode reproduzir o conjunto identicamente.

## 6. Como reproduzir

**Pré-requisitos:**

- S1 concluído (ambiente Conda `tcc` funcional)
- Cadastro no fastMRI completo (https://fastmri.med.nyu.edu/) e e-mail de aprovação recebido
- Pelo menos 200 GB livres em algum disco
- Conexão de internet com ~50 Mbps (tempo total de download ~12 h)

**Passo 1: obter URLs assinadas do fastMRI.**

Após aprovação, o fastMRI envia um e-mail com URLs assinadas para os tarballs. Para o `multi_train` há 9 tarballs (`brain_multicoil_train_part_1.tar.gz` até `_part_9.tar.gz`). As URLs são válidas por 24h.

**Passo 2: clonar fastMRI+ e obter `brain.csv`.**

```bash
git clone https://github.com/microsoft/fastmri-plus.git data/fastmri-plus
cp data/fastmri-plus/Annotations/brain.csv \data\mri\brain.csv
```

**Passo 3: download sequencial.**

Para cada tarball N de 1 a 9:

```bash
# 1. Baixar (no Git Bash ou similar)
wget "URL_ASSINADA_TARBALL_N" -O brain_multicoil_train_part_${N}.tar.gz

# 2. Extrair em pasta temporária
mkdir -p \data\mri\multi_train_temp
tar -xf brain_multicoil_train_part_${N}.tar.gz -C \data\mri\multi_train_temp

# 3. Filtrar para \data\mri\anotados\
python scripts/filtra_anotados.py \
    --origem \data\mri\multi_train_temp \
    --destino \data\mri\anotados \
    --brain-csv \data\mri\brain.csv

# 4. Limpar
rm -rf \data\mri\multi_train_temp
rm brain_multicoil_train_part_${N}.tar.gz
```

**Saída esperada após o tarball 9:** ~750 arquivos em `\data\mri\anotados\`. Distribuição por sequência (verificável com `python scripts/audit_dataset.py`) deve ser próxima de 447/340/210 para AXFLAIR/AXT1/AXT1POST.

**Passo 4: verificar integridade.**

```bash
python scripts/audit_dataset.py
```

Saída esperada (estado pós-S2, antes do refilter de S3):

```
Volumes em disco: 750
Linhas no brain.csv: 8213
Volumes COM bbox slice-level: 352
Volumes SO com study-level (sem bbox): 398
Volumes em disco mas fora do brain.csv: 0
```

Se Cobertura ≠ 750 ou houver volumes fora do `brain.csv`, a fase não foi reproduzida fielmente.
